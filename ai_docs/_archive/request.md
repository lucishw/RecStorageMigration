# [Project Prompt] Multi-Worker Storage Migration Manager (.NET 4.6.1 / WinForm)

## 1. 개요 (Overview)
- 현재 프로젝트는 파일 추적 및 암호화를 위한 기능으로 개발된 프로젝트입니다.
- 현재 프로젝트의 화면 및 프로그램 구조 (배치 동작) 을 기반으로 하여 대규모 스토리지(약 130TB, 약 1억 건)에 저장된 파일들을 DB 데이터를 기반으로 추적하여 최종 결과 폴더로 복사하고, 처리 결과를 원본 DB에 실시간 마킹하는 WinForm 프로그램을 만들고자 합니다.
- 기존에는 단일 스레드로 구현되어 여러 조건별로 프로세스(.exe)를 여러 개 띄워 수동 관리했으나, 관리 편의성과 성능 최적화를 위해 하나의 WinForm 내에서 멀티 스레드로 제어하는 '멀티 작업 관리자(Multi-Worker Manager) 패턴'으로 리팩토링하고자 합니다.

## 2. 현재 시스템 및 인프라 현황 (Current Status & Infrastructure)
- **개발 환경:** C# .NET Framework 4.6.1, WinForm 익스텐션 기반으로 개발되어 있으나, 성능적으로 개선이 분명하며 영향도가 없다면 .NET 버전 업그레이드가 가능합니다.
- **스토리지 아키텍처:** 네트워크로 연결된 독립된 분산 스토리지(NAS) 구조입니다. 대규모 트리 구조(5 depth, Deep Tree Structure)로 파일이 분산 저장되어 있습니다.
- **주요 제한 사항 (중요):** 
  1. NAS 특성상 상위 디렉터리에 대한 대규모 탐색(`Directory.GetFiles` 등) 시 극심한 파일 I/O 지연 및 타임아웃이 발생하므로 **디렉터리 스캔 방식은 사용할 수 없습니다.**
  2. 중간에 시스템이 정전, 강제 종료되더라도 재시작 시 멈춘 지점부터 안전하게 이어 갈 수 있도록 **반드시 실시간(건별, Row by Row)으로 DB 마킹(Update)이 즉시 반영되어야 합니다.** 하지만, 추가적으로 다른 효율적인 방안이 있다면 사용자에게 제안을 할 수 있습니다.

## 3. UI 및 프로그램 동작 시나리오 (UI & Implementation Scenario)
1. **스레드 가용 개수 인식:** 프로그램 로드 시 시스템의 논리 프로세서(스레드) 개수를 자동으로 읽어옵니다.
2. **동적 UI 생성:** 읽어온 가용 스레드 개수만큼 WinForm 내 `FlowLayoutPanel` 등에 스레드별 설정 항목 행(Row)을 동적으로 생성합니다. (물론 사용자가 직접 row 수를 관리 가능해야합니다.)
3. **사용자 정의 조건 부여:** 각 행별로 사용자가 직접 독립된 제어 환경을 설정합니다.
   - 활성화 여부 (Check 옵션)
   - DB 조회 조건 필터 (Where 절 텍스트: 날짜 범위, 업무 코드 등 지정을 위함, text)
   - Read Top Count (text, default 5000)
   - 마킹 컬럼명 및 마킹 처리 값 (Value)
   - 개별 스레드 시작 / 중지 버튼 및 진행률/상태 표시 라벨
4. **개별 독립 구동:** 각 작업 스레드는 시작 시 사용자가 부여한 개별 조건에 맞춰 아래 로직을 데이터 소진 시까지 독립 실행합니다.
   - 각 개별 독립 구동하는 현황을 실시간으로 파악하기 위해, 현황 파악 버튼을 클릭하며 현재까지 처리된 범위에 대해서 전체건수 기준 처리된 건수 (다양한 예외케이스 건수)를 보여주도록 기능을 추가합니다. 그래프로 날짜 추이도 확인이 가능해야 합니다.

## 4. 핵심 비즈니스 로직 프로세스 (Core Core Logic per Worker Thread)
1. **DB 분할 조회:** 부여받은 개별 Where 조건과 마킹 조건이 반영된 쿼리를 통해 대상 데이터를 읽어옵니다. (ex. `SELECT TOP 1000 ... WHERE Status = 0 AND [사용자조건]`)
2. **데이터 처리 및 경로 확인:** 1,000건의 row를 내부 루프로 돌며 데이터 내부 정보를 조합해 소스 파일 경로를 생성합니다.
3. **네트워크 I/O 최적화 (중요):** 네트워크 지연(Latency)을 최소화하기 위해 `File.Exists()` 검증 과정을 건너뛰고, **즉시 `File.Copy()`를 시도**합니다. 파일이 없는 비정상 데이터는 `try-catch` 블록에서 `FileNotFoundException` 예외로 잡아 안전하게 예외 처리합니다.
4. **즉시 DB 마킹:** 파일 복사 성공 혹은 예외 발생 시, 원본 DB 테이블에 처리 상태를 즉시 `UPDATE`하여 장애 지속성을 확보합니다.

## 5. 개발 시 필수 주의사항 및 기술 제약 (Crucial Technical Constraints)
코드를 작성할 때 다음 문제를 반드시 예방하도록 아키텍처를 설계해 주세요:
- **주석 필수** 모든 코드를 수정 및 적용하기 위해서는 반드시, 해당 코드에 대한 이해가 용이하도록 주석을 작성해주세요. 주석은 다음과 같은 형태로 어미를 정리합니다. eg. 처리 기능 함수입니다. X / 처리 기능 함수 O 
- **코드 단순화** 모든 코드를 수정할 때에는 기존의 함수를 최대한 활용하며, 골격 구조를 유지하도록 하며 가능한 이해하기 쉬운 코드로 수정을 진행해주세요. 함수명도 단순화하여 너무 길어지지 않도록 하되, 변수명은 상세하게 작성해주세요. 만약 구조의 변경이 불가피하게 필수적으로 필요한 경우 사용자에게 반드시 영향도를 파악 후 상세하게 정리하여 내용을 알리고, 컨펌을 받도록 해주세요.
- **DB 커넥션 충돌 방지:** 모든 스레드가 단일 `SqlConnection`을 공유하면 안 됩니다. 각 스레드 루프 내부, 혹은 스레드 인스턴스마다 **독립된 DB 커넥션을 열고 닫도록(`using`문 활용)** 격리해 주세요.
- **크로스 스레드(Cross-Thread) 차단:** 백그라운드 스레드에서 개별 UI 항목(라벨 상태, 텍스트박스 등)의 값을 직접 수정하면 프로그램이 즉시 크래시됩니다. UI 컨트롤을 업데이트할 때는 반드시 `this.Invoke` 또는 `this.BeginInvoke`를 사용하여 메인 UI 스레드에 제어권을 위임해 주세요.
- **.NET 네트워크 연결 제한 해제:** 동일한 원격 서버(NAS/DB)로의 동시 커넥션 수 제한을 풀기 위해, 폼 초기에 `System.Net.ServicePointManager.DefaultConnectionLimit = 100;` 설정을 반드시 선언해 주세요.
- **UI 비응답(화면 멈춤) 방지:** DB 조회 및 파일 I/O 작업 동안 UI 스레드가 블로킹되지 않도록 `Task.Run` 또는 비동기(`async/await`) 패턴을 적절히 활용하여 메인 화면이 '응답 없음'이 되지 않게 구현해 주세요.
- **파일 구조** frmMain.cs 안에 모든 클래스나 함수를 넣지 말아주세요. 독립적으로 재활용이 가능한 함수를 생성한 경우에는 추가 클래스 파일을 생성하여 유지보수가 용이하도록 해주세요.
- **프로젝트명 변경** 기능의 변경에 따라, 프로젝트명을 RecFileMigration 으로 변경해주세요.
