# 운영자/엔지니어 관점 UI/UX 요구사항

## 목표

UI를 개발자 관점이 아니라 운영자와 엔지니어 관점에서 설계한다.

## 운영자가 즉시 답할 수 있어야 하는 질문

- 전체 작업이 얼마나 진행되었는가?
- 현재 정상적으로 처리되고 있는가?
- 병목은 DB인가, 파일 I/O인가, 네트워크인가?
- 실패한 파일은 몇 건인가?
- 실패 원인은 무엇인가?
- 어떤 Worker에서 문제가 발생했는가?
- 현재 네트워크 대역폭은 얼마나 사용 중인가?
- 남은 작업량은 어느 정도인가?
- 예상 종료 시각은 언제인가?
- 재처리가 필요한 대상은 무엇인가?
- DB Bulk Update는 정상적으로 수행되고 있는가?
- Work Queue와 Result Queue가 정상적으로 소진되고 있는가?

## 화면 우선순위

### 1순위: 전체 상태

- 전체 진행률
- 성공/실패/대기 건수
- 현재 처리 속도
- 예상 남은 시간

### 2순위: 병목 파악

- Network 사용량
- DB Bulk Update 대기 건수
- Work Queue 대기 건수
- CPU/Memory/Disk I/O

### 3순위: Worker 상태

- Worker별 상태
- Worker별 처리 속도
- Worker별 오류

### 4순위: 오류 대응

- 최근 오류 목록
- 실패 파일 목록
- 재시도 가능 여부
- 오류 상세 메시지

## 운영 액션

UI에는 다음 운영 액션을 고려한다.

- 작업 시작
- 작업 일시정지
- 작업 재개
- 작업 중지
- 실패 건 재처리
- Bulk Update 강제 Flush
- Worker 수 변경
- Bandwidth Limit 변경
- 오류 로그 Export

## Cursor 지침

```md
UI를 설계할 때 운영자와 엔지니어가 실제 운영 중 어떤 정보를 먼저 확인해야 하는지 기준으로 화면을 구성한다.

운영자는 전체 진행률, 처리 속도, 실패 건수, 병목 지점, Worker 상태, Network 사용량, 재처리 대상을 즉시 확인할 수 있어야 한다.

대시보드에는 단순 로그보다 요약 지표와 실시간 상태를 우선 배치한다.
```
