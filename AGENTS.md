# Repository Guidelines

## Project Structure & Module Organization

This repository contains a single Visual Studio solution, `RecStorageMigration.sln`, with one C# WinForms project in `KMSMigration/`. Application startup is in `KMSMigration/Program.cs`; the main form lives in `frmMain.cs`, `frmMain.Designer.cs`, and `frmMain.resx`. Shared helpers are grouped under `KMSMigration/Common/` for database, file, INI, logging, SQL account, and KMS/media operations. Project resources and assembly metadata are in `KMSMigration/Properties/`. Build output, runtime DLLs, logs, and local INI files appear under `KMSMigration/bin/` and `KMSMigration/obj/`; avoid editing generated files there unless you are updating packaged runtime assets intentionally.

## Build, Test, and Development Commands

Use a Visual Studio Developer Command Prompt or another shell where MSBuild is available.

```powershell
msbuild RecStorageMigration.sln /p:Configuration=Debug /p:Platform=x86
msbuild RecStorageMigration.sln /p:Configuration=Release /p:Platform=x64
.\KMSMigration\bin\Debug\RecStorageMigration.exe
```

The solution targets .NET Framework 4.6.1 and defines Debug/Release builds for x86 and x64. The project references `MediaParser.dll` from `bin\Debug`, so verify required native DLLs are present before running locally.

## Coding Style & Naming Conventions

Follow the existing C# style: four-space indentation, braces on new lines, PascalCase for classes and public methods, and descriptive method names such as `KMSDecryptFile`. Keep WinForms designer changes in `.Designer.cs` generated by Visual Studio whenever possible. Put reusable logic in `Common/` rather than in form event handlers. Preserve existing Korean comments where they clarify UI or runtime behavior.

## Testing Guidelines

No automated test project is currently present. For changes, at minimum build the affected configuration and perform a manual WinForms smoke test. If adding tests, create a separate `RecStorageMigration.Tests/` project using MSTest, NUnit, or xUnit, name tests after the behavior under test, and keep fixtures independent of local `bin/` logs or INI files.

## Commit & Pull Request Guidelines

This checkout does not expose local Git history, so no repository-specific commit convention can be inferred. Use concise imperative commit subjects, for example `Fix KMS initialization error handling`. Pull requests should describe the user-visible change, list tested build platforms, note any required DLL or configuration updates, and include screenshots for UI changes.

## Security & Configuration Tips

Do not commit real credentials, server IPs, or customer data in `app.config`, `ini/`, logs, or backup folders. Treat generated logs and media backup files under `bin/` as local runtime artifacts unless explicitly needed for release packaging.

## Agent-Specific Instructions

Before modifying code, summarize the planned scope, exact change details, and expected impact for the user. Start code edits only after the user explicitly responds with `수정 승인`.

Before deleting code, summarize the deletion scope, removed behavior, and expected impact for the user. Delete code only after the user explicitly responds with `삭제 승인`.

When changing code, use intuitive function names and prefer step-by-step, readable implementation over compact use of newer language syntax.

Keep changes within the existing structural shape of the code by default. If a structural refactor would provide substantial benefits, explain the benefits and impact to the user first, then wait for `수정 승인` before starting any refactor work.
